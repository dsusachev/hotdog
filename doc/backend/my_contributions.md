# Мой вклад в проект — Backend

## О проекте

**Hotdog / FoodScanner** — веб-сервис распознавания продуктов питания по фотографии.  
Пользователь загружает фото → ML-модель определяет продукт → сервис показывает цены в магазинах, рецепты, состав.

**Команда:** 6 ролей — product owner, frontend, ML, database, testing, backend.  
**Моя роль:** backend (не связан с ML и database).

**Стек:** Python, FastAPI, PostgreSQL (Supabase), SQLAlchemy (async), JWT, httpx, Docker.

---

## Что я сделал

### 1. REST API — 10 эндпоинтов

Спроектировал и реализовал всё API приложения:

| Эндпоинт | Описание |
|---|---|
| `POST /api/auth/register` | Регистрация пользователя |
| `POST /api/auth/login` | Вход, выдача JWT-токена |
| `GET /api/auth/me` | Профиль текущего пользователя |
| `POST /api/classify` | Загрузка фото и распознавание продукта |
| `GET /api/products` | Поиск продуктов через Open Food Facts |
| `GET /api/prices` | Цены и ближайшие магазины по координатам |
| `GET /api/geocode` | Координаты → человекочитаемый адрес |
| `GET /api/history` | История запросов пользователя |
| `POST /api/feedback` | Отзыв пользователя |
| `GET /api/recipes` | Рецепты через TheMealDB |
| `GET /api/categories` | CRUD-справочник категорий |
| `GET /health` | Статус сервиса (для мониторинга) |

Обеспечил единый формат ответа для всех ошибок — фронтенд всегда получает предсказуемую структуру `{"status": "error", "code": ..., "message": ...}`.

---

### 2. JWT-аутентификация

- Хэширование паролей через **bcrypt** — алгоритм, специально спроектированный быть медленным и стойким к брутфорсу
- Асинхронная обёртка над bcrypt через `run_in_executor`, чтобы CPU-тяжёлая операция не блокировала весь сервер
- Создание и верификация JWT-токенов (HS256, срок жизни 24 часа)
- Dependency injection `getCurrentUser` / `getCurrentUserOptional` — FastAPI автоматически проверяет токен перед каждым защищённым эндпоинтом

---

### 3. Настройка подключения к базе данных

- Асинхронное подключение к PostgreSQL через SQLAlchemy + asyncpg
- Решил нетривиальный баг: на Linux asyncpg зависал при TLS-negotiation с Supabase. Добавил явный SSL-контекст, который пропускает negotiation и сразу идёт в TLS-handshake
- Graceful degradation при старте: если БД недоступна — сервер не падает, а логирует предупреждение и продолжает работу

---

### 4. Интеграция с ML-сервисом

- Клиент к ML-сервису (отдельный микросервис на PyTorch, сделанный ML-командой)
- Валидация загружаемого изображения в 5 шагов: наличие файла, расширение, пустой файл, размер (макс. 5 МБ), целостность через Pillow
- Fallback на mock-ответ если ML-сервис недоступен, с флагом `"mock": true` — фронтенд показывает предупреждение, сервис остаётся рабочим
- Автоматическое сохранение каждой классификации в историю пользователя

---

### 5. Интеграции с внешними API

Написал клиентов к четырём внешним сервисам:

| Сервис | Что делает |
|---|---|
| **Open Food Facts** | Поиск продуктов, КБЖУ. Retry-логика: повтор при таймауте, без повтора при HTTP-ошибке |
| **Open Prices** | Цены на продукты по названию или штрихкоду |
| **Nominatim (OpenStreetMap)** | Обратное геокодирование: координаты → адрес |
| **Overpass API (OSM)** | Ближайшие магазины, кафе, рестораны в радиусе 2 км |
| **TheMealDB** | Рецепты по категории или названию |

Везде обработаны сценарии недоступности: таймауты, HTTP-ошибки, невалидный JSON — возвращается пустой результат вместо падения сервера.

---

### 6. Middleware

**Logging middleware** — оборачивает каждый запрос:
```
→ POST /api/classify
← POST /api/classify — 200 (312ms)
```
Ключевой инструмент отладки в командной разработке: при баг-репорте от фронтенда сразу видно какой запрос пришёл, что ответил сервер и за сколько.

**Три глобальных обработчика ошибок:**
- HTTP-ошибки (401, 404 и т.д.)
- Ошибки валидации входных данных (422) — с указанием конкретных полей
- Любое необработанное исключение → 500 без стектрейса наружу

---

### 7. Оркестратор запуска (`start.py`)

Скрипт, который одной командой `python start.py` запускает все три сервиса параллельно:
- ML-сервис → `http://localhost:8001`
- Backend API → `http://localhost:8000`
- Frontend → `http://localhost:3000`

При первом запуске автоматически скачивает ML-модель (~90 МБ) из GitHub Releases. Если torch не установлен — запускает stub-модель, сайт остаётся рабочим. Ctrl+C корректно останавливает все три процесса.

---

## Роль в командном взаимодействии

Backend находился на стыке трёх команд:

- **Фронтенд** ждал стабильный API-контракт. Я обеспечил единый формат ошибок и mock-режим ML, чтобы фронтенд мог работать независимо от ML-сервиса
- **ML-команда** делала отдельный сервис. Я написал клиента к нему с fallback, чтобы недоступность ML не роняла весь сайт
- **DB-команда** делала схему базы данных. Я писал SQLAlchemy-модели и запросы поверх этой схемы

Благодаря mock-режимам и graceful degradation каждая команда могла работать и тестировать свою часть независимо от остальных.
