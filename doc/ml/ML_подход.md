# ML-подход

> Документ описывает машинно-обучающую часть проекта по классификации продуктов питания по фотографии. Покрывает: постановку задачи, выбор данных, архитектуру, стратегию обучения, обработку «неизвестных» объектов, метрики и план деплоя.

---

## 1. Постановка задачи

**Тип задачи:** multi-class single-label классификация изображений.

**Вход:** изображение продукта (JPG/PNG), загруженное пользователем — либо снятое самостоятельно, либо скачанное из интернета.

**Выход:** категория продукта + уверенность модели + top-k альтернатив + флаг «неизвестный объект».

**Целевые условия использования:** реальные фотографии в неконтролируемых условиях — разное освещение, фон, ракурсы, качество камеры.

---

## 2. Скоуп MVP

**Включено:**
- Фрукты и овощи в естественном виде.
- Упакованные супермаркетные товары (молочка, бакалея, снеки и т.п.).

**Исключено из MVP:**
- Готовые блюда (pizza, sushi, pasta и т.д.) — требуют отдельного датасета (например, Food-101) и более сложной модели; вынесены в потенциальное расширение.
- Composite-фото с несколькими продуктами в кадре — модель работает в режиме single-label.

**Логика выбора скоупа:** фрукты/овощи и упакованные товары имеют чёткие визуальные признаки (форма, цвет, упаковка), что делает задачу решаемой при ограниченных ресурсах и в рамках учебного проекта.

---

## 3. Стратегия работы с данными

**Принципиальный выбор:** один реалистичный grocery-датасет в качестве основного. Конкретный выбор датасета — отдельная задача (см. отдельный документ).

**Обоснование выбора реалистичного датасета** (а не «лабораторного» вроде Fruits-360 с белым фоном):

- Inference-домен — фото в реальных условиях, поэтому датасет должен соответствовать домену.
- Лабораторные датасеты дают красивые метрики на test set, но плохо переносятся в продакшен (известная проблема domain shift).
- Один источник данных проще для отладки, для отчётности, и не требует ручного маппинга классов между датасетами.

**Разделение данных:** train / validation / test в пропорции 70 / 15 / 15.

- Stratified split — обязательно, чтобы пропорции классов сохранялись во всех наборах.
- Test set фиксируется один раз и не используется ни для подбора гиперпараметров, ни для подбора порога — только для финальной оценки.

**Дополнительный датасет** (для проверки робастности модели вне основного распределения) — отдельная задача. Будет использован только для evaluation, без обучения.

## 4. Архитектура модели

**Основная модель:** **EfficientNet-B0** (предобученная на ImageNet).

**Обоснование:**

| Критерий | EfficientNet-B0 |
|---|---|
| Параметров | ~5.3M |
| Размер весов | ~20 MB |
| Inference на CPU | десятки мс на изображение |
| ImageNet top-1 | ~77% |
| Доступность весов | `torchvision.models` / `timm` |

EfficientNet разработан Tan & Le (2019) специально для resource-constrained сценариев через compound scaling — это стандартный выбор для задач с ограниченным compute и требованием инференса без GPU. Для нашего сценария (классификация на ~25–80 классов, инференс на сервере без GPU) — оптимальный баланс качества и стоимости.

**Baseline для сравнения:** **ResNet50** (предобученная на ImageNet).

**Зачем нужен baseline:** для честной защиты выбора архитектуры. Сравнение «EfficientNet-B0 vs ResNet50» покажет, что выбор сделан осознанно, а не «потому что популярна».

**Что не использовалось и почему:**
- **ViT** — требует больше данных для эффективного fine-tuning, сложнее в настройке для junior-уровня.
- **DenseNet-161** — тяжёлая, оверкилл для задачи.
- **MobileNetV3** — оптимизирована под мобильный инференс, у нас серверный.

## 5. Transfer learning стратегия

**Подход:** двухэтапный fine-tuning.

### Этап 1. Feature extraction (warm-up)

- Backbone полностью заморожен (`requires_grad = False`).
- Финальный classifier-слой заменён на новую голову под количество целевых классов.
- Обучается только новая голова.
- LR: `1e-3`.
- Эпох: 3–5.
- Цель: дать новой случайно инициализированной голове найти разумную начальную точку, не повредив предобученные веса backbone.

### Этап 2. Fine-tuning

- Размораживаются верхние блоки backbone (для EfficientNet-B0 — последние 2–3 блока, не весь backbone сразу).
- LR: `1e-4` или `1e-5` (на порядок-два меньше, чем на этапе 1).
- Эпох: 10–20.
- Цель: адаптировать high-level признаки под специфику продуктов, сохранив низкоуровневые фильтры.

**Почему два этапа, а не один:** случайно инициализированная голова в начале обучения генерирует большие градиенты, которые при разблокированном backbone «откатывают» предобученные веса в случайное состояние. Это destructive interference — известная проблема transfer learning, и двухэтапный подход её предотвращает.

### Особенность работы с BatchNorm на этапе 1

EfficientNet-B0 содержит BN-слои внутри. При заморозке параметров (`requires_grad = False`) обучаемые параметры BN (gamma, beta) перестают обновляться через градиенты — это правильно. Но running statistics (mean, var) обновляются в forward pass при `model.train()` режиме, и могут «уплыть» под распределение нового датасета даже при замороженных параметрах.

**Решение:** на этапе 1 backbone явно переводится в `eval()` режим:

```python
model.train()           # вся модель в train
model.backbone.eval()   # backbone в eval — BN использует ImageNet running stats
```

На этапе 2 переключаемся обратно в `train()` для всей модели — теперь BN-статистики должны адаптироваться к нашим данным.

## 6. Параметры обучения

| Параметр | Значение |
|---|---|
| Optimizer | AdamW |
| Loss | CrossEntropyLoss + label smoothing (0.1) |
| LR scheduler | CosineAnnealingLR (применяется отдельно на каждом этапе) |
| Weight decay | 1e-4 (стандарт для AdamW) |
| Batch size | 32–64 (зависит от памяти Colab) |
| Mixed precision | Да (`torch.cuda.amp`) — ускорение в 1.5–2× на T4 |
| Gradient clipping | `max_norm=1.0` (страховка от взрывающихся градиентов) |
| Early stopping | patience = 5–7 эпох по validation loss |

**Что не используется и почему:**

- **LR warm-up** — критичен для обучения трансформеров с нуля или при больших batch size, для fine-tuning EfficientNet с маленьким LR не даёт значимого эффекта. Усложнение без выгоды.
- **SGD с momentum** — рассматривался, но AdamW проще в подборе гиперпараметров и стандартен для transfer learning.

## 7. Preprocessing и augmentations

### Preprocessing (одинаковый для train/val/test)

- Resize до 224×224 (стандартный input EfficientNet-B0).
- Convert to RGB tensor.
- Normalize с ImageNet mean/std:
  - `mean = [0.485, 0.456, 0.406]`
  - `std = [0.229, 0.224, 0.225]`

Нормализация **обязательна** — модель предобучена на нормализованных данных, без неё на выходе будет мусор.

### Train augmentations

| Аугментация | Параметры | Смысл |
|---|---|---|
| `RandomResizedCrop` | `(224, scale=(0.7, 1.0))` | Робастность к разному масштабу и кадрированию |
| `RandomHorizontalFlip` | `p=0.5` | Продукт узнаваем зеркально |
| `ColorJitter` | `brightness=0.3, contrast=0.3, saturation=0.3, hue=0.1` | Разное освещение |
| `RandomRotation` | `degrees=15` | Небольшие повороты камеры |
| `RandomPerspective` | `p=0.3` | Лёгкие искажения от ракурса |
| `GaussianBlur` (опц.) | `p=0.2` | Робастность к плохим фото |

### Validation/Test augmentations

Только resize и normalization, без случайностей — иначе метрики будут шумные и невоспроизводимые.

### Что намеренно не используется

- `RandomVerticalFlip` — упаковка молока вверх ногами это другой объект.
- Слишком агрессивный `ColorJitter` — банан станет зелёным, модель запутается.

### Регуляризация — Dropout

EfficientNet-B0 уже содержит dropout перед classifier (`p=0.2`). В новой голове добавляется ещё один:

```python
classifier = nn.Sequential(
    nn.Dropout(p=0.3),
    nn.Linear(1280, num_classes)
)
```

Если данных будет много (десятки тысяч изображений) — рассмотреть голову с промежуточным слоем:

```python
classifier = nn.Sequential(
    nn.Linear(1280, 512),
    nn.ReLU(),
    nn.Dropout(p=0.4),
    nn.Linear(512, num_classes)
)
```

Dropout внутри backbone не добавляется — он там уже есть, где нужно.

## 8. Threshold-логика для unknown

**Проблема:** модель всегда что-то предсказывает — softmax выдаст распределение по классам даже на изображении кота. Без отсечения unknown продукт будет уверенно говорить «это банан» на любую ерунду, что убьёт доверие пользователей.

**Подход:** softmax thresholding.

```python
probs = F.softmax(logits, dim=1)
max_prob, pred_class = probs.max(dim=1)
if max_prob < THRESHOLD:
    return {"category": None, "is_unknown": True, "confidence": max_prob}
else:
    return {"category": pred_class, "is_unknown": False, "confidence": max_prob}
```

**Подбор порога (экспериментальный, на validation set):**

1. Прогнать модель на validation set, для каждого изображения сохранить max_prob и факт правильности.
2. Построить распределение max_prob отдельно для правильных и неправильных предсказаний.
3. Выбрать порог, отсекающий большинство неправильных предсказаний с минимальной потерей правильных. Обычно 0.5–0.7.
4. Дополнительно: подготовить набор из ~50–100 заведомо unknown изображений (кошки, машины, случайные объекты) и измерить, какой процент модель отсекает при выбранном пороге.

**Известные ограничения подхода:**

Softmax confidence — это не настоящая вероятность правильности предсказания. Нейросети после fine-tuning склонны быть overconfident, особенно на out-of-distribution данных. Softmax threshold — это простой baseline, для более серьёзных задач существуют MC Dropout, Energy-based scores, ODIN. Для учебного проекта softmax threshold достаточен; продвинутые методы вынесены в раздел future work.

**Опциональное улучшение:** **temperature scaling** — простая калибровка, делящая логиты на температуру `T`, найденную на validation set. Делает confidence менее overconfident. Реализация ~30 строк, рекомендуется при наличии времени.

## 9. Метрики оценки

### Основные метрики качества классификации

- **Top-1 accuracy** — доля правильных top-1 предсказаний. База.
- **Top-3 accuracy** — доля случаев, когда правильный класс в топ-3. Прямое отражение качества пользовательского опыта (в API возвращается top-k).
- **Macro F1** — среднее F1 по всем классам без учёта частоты. Честная метрика для несбалансированных датасетов.
- **Per-class precision / recall / F1** — таблица по каждому классу. Показывает, какие классы модель путает.
- **Confusion matrix** — визуализация per-class метрик через `sklearn.metrics.ConfusionMatrixDisplay` или seaborn heatmap.

### Метрики для threshold-логики

- **Coverage** — доля изображений, на которых модель уверена (max_prob ≥ threshold).
- **Selective accuracy** — accuracy только на «уверенных» изображениях. Должна быть выше общей.
- **OOD detection rate** — % правильно отклонённых на ручном unknown-наборе.

### Что показывается в финальном отчёте

1. Сводная таблица: accuracy / top-3 / macro F1 на validation и test.
2. Confusion matrix как изображение.
3. Топ-5 лучших и топ-5 худших классов по F1 — с интерпретацией.
4. График threshold vs (coverage, selective accuracy) — обоснование выбранного порога.
5. Примеры успешных и провальных предсказаний.

### Обязательные принципы

- Test set используется **только** для финальной оценки, не для подбора порога или гиперпараметров.
- Метрики приводятся **раздельно** для validation и test, чтобы видеть переобучение.
- Для несбалансированных классов смотреть **macro F1**, а не weighted — weighted даёт оптимистичную картину.

## 10. Inference pipeline и деплой

### Pipeline одного предсказания

1. Получение изображения из HTTP-запроса (multipart/form-data).
2. Валидация: формат (JPG/PNG), размер, целостность файла.
3. Декодирование и конвертация в RGB.
4. Preprocessing (resize, normalize).
5. Forward pass через модель.
6. Softmax + извлечение top-k.
7. Применение threshold-логики.
8. Формирование JSON-ответа.

### Формат ответа

Детально проектируется в отдельной задаче. Ключевые поля:

- `category` — предсказанный класс (или `null` при `is_unknown=True`).
- `confidence` — max softmax probability.
- `top_k` — массив альтернативных предсказаний.
- `is_unknown` — флаг неуверенного предсказания.

### Деплой-архитектура

ML-сервис вынесен в **отдельное приложение** (FastAPI), которое запускается отдельным процессом. Основной backend ходит к нему по HTTP.

**Преимущества:**
- Модель загружается в память один раз при старте.
- Backend и ML деплоятся независимо.
- ML-сервис при необходимости можно вынести на отдельную машину (например, с GPU).
- Чистое разделение ответственности.

### Compute-план

- **Обучение:** Google Colab Free (T4 GPU). При нехватке лимитов — Colab Pro.
- **Локальная разработка / отладка / инференс:** MacBook M1 через PyTorch MPS backend.
- **Чекпоинты:** сохраняются на Google Drive после каждой эпохи (Colab-сессии нестабильны).
- **Логирование экспериментов:** TensorBoard или Weights & Biases.

### Стек

- Python 3.10+
- PyTorch + torchvision
- (Опц.) timm — расширенная коллекция предобученных моделей
- FastAPI — ML-сервис
- Pillow / OpenCV — работа с изображениями
- scikit-learn — метрики и confusion matrix
